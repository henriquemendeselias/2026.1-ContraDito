## 1. Contexto e Problema

A arquitetura assíncrona baseada em RAG (estabelecida na ADR 001) resolveu o problema de precisão semântica da IA, mas manteve um acoplamento crítico na camada de execução.

O backend principal (FastAPI) ainda atuava como orquestrador direto das requisições pesadas de IA, gerando os seguintes riscos estruturais:

### 1.1 Risco de Indisponibilidade Geral (Timeouts e Out of Memory)

Manter a API web esperando ou gerenciando tarefas pesadas de processamento local (execução do Llama 3.1 8B e geração de embeddings com SBERT) abria margem para esgotamento de memória RAM/VRAM.

Um travamento ou lentidão no processamento de madrugada derrubaria o portal de consulta dos usuários.

### 1.2 Sobrecarga de Abstração no Código

A equipe debateu a adoção da Arquitetura Hexagonal (Ports and Adapters) para blindar o domínio. Contudo, para a realidade de prazos do projeto e visando agilidade na entrega, essa abordagem introduziria complexidade desnecessária, dificultando o rastreamento e a lógica de cálculo do Score.

### 1.3 Pontos Cegos na Modelagem de Dados

A estrutura relacional anterior não comportava regras de negócio cruciais, como:

- A barreira temporal de discursos (RN05);
- A troca de partidos por parlamentares no decorrer da legislatura;
- A exibição de metadados de interface no front-end  
  _(ex: data de última atualização do score)_.

---

## 2. Decisão Arquitetural

Para garantir resiliência absoluta e simplicidade de código, decidimos adotar duas abordagens arquiteturais complementares.

### 2.1 Macroarquitetura: CQRS (Command Query Responsibility Segregation)

O sistema foi dividido física e logicamente em dois serviços independentes que não realizam chamadas HTTP diretas entre si, utilizando o Supabase (PostgreSQL + pgvector) como único meio de comunicação.

#### Lado de Leitura (Query — FastAPI)

Uma API restrita estritamente a consultas rápidas.

Responsabilidades:

- Ler dados já consolidados;
- Validar componentes fechados (dropdowns);
- Entregar JSONs ao Next.js em milissegundos;
- Utilizar cache em memória.

A FastAPI é completamente cega para os modelos de IA.

#### Lado de Escrita/Processamento (Command — Worker NLP)

Serviço Python isolado de background executado via rotinas agendadas (Cron Jobs).

Responsabilidades:

- Interagir com SBERT;
- Executar o Llama 3 localmente;
- Consumir o processamento pesado;
- Persistir os vereditos e scores no banco.



### 2.2 Microarquitetura do Worker: Pipe and Filter.

Afastamos o modelo Hexagonal no design de código do Worker.

O fluxo interno de processamento assíncrono operará estritamente como um pipeline procedural e linear.

O pacote de dados trafega de forma determinística por 5 filtros sequenciais:

```text
Filtro 1 (Extração de dados da API)
    ↓
Filtro 2 (Resumo Executivo pelo Llama)
    ↓
Filtro 3 (Geração dos chunks dos discursos extraídos)
    ↓
Filtro 4 (Vetorização do Resumo Executivo e dos chunks de discurso via SBERT)
    ↓
Filtro 5 (Recuperação de Contexto RAG)
    ↓
Filtro 6 (Veredito Llama 3.1 e Cálculo do Score)
```


## 3. Consequências

### Pontos Positivos

#### Resiliência e Tolerância a Falhas

Se o modelo Llama 3 sofrer timeout, a falha fica restrita ao contêiner isolado do Worker.

A FastAPI continua funcionando normalmente no ambiente web, servindo os dados consolidados da última execução bem-sucedida.

#### Manutenibilidade Simplificada

O pipeline linear (Pipe and Filter puro) no Worker facilita o debug.

Se o score de um político divergir, a equipe consegue isolar o erro testando as funções sequenciais passo a passo no código Python, sem camadas de abstração complexas.

#### Leitura de Alta Performance

A FastAPI atua com queries SQL diretas e leves sobre tabelas indexadas.

O tempo de resposta para o Next.js cai para a escala de milissegundos, pois nenhum cálculo vetorial ou inferência de linguagem ocorre em tempo de execução do usuário.

### Trade-offs

#### Consistência Eventual

Os dados do portal não são exibidos em tempo real.

O score do político só muda após o fechamento da esteira do Worker de madrugada.

#### Necessidade de Invalidação Ativa

Exige o disparo de um comando administrativo simples do Worker para a FastAPI ao término do processamento pra limpar o cache em memória da API e disponibilizar instantaneamente os dados recém-processados.
